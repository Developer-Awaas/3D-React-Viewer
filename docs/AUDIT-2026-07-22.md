# Drishti Full-Product Audit — 22 Jul 2026 (pre-launch)

Three parallel deep audits: (1) ML pipeline & parsing engine, (2) features + frontend, (3) Supabase/limits/edge cases. Every finding has file+function references. Verified independently: Vastu N/S orientation is CORRECT (scene_builder.py L66 and pdf_vector.py L1200 both flip to bottom-left origin, y-up; vastu.py atan2 math checks out).

## VERDICT IN ONE LINE
The vector-parsing engine is strong and defensively written. Value is leaking in three places: the ML readers (winner-takes-all, tf2 door bug, room-types discarded), the render path (seg/depth "moat" mostly not fed, missing free SDXL speed knobs), and launch security (no /review auth, rate limiter dead behind the tunnel).

---

## A. LAUNCH BLOCKERS — fix in the next 4 hours (all SMALL except noted)

1. **/review has NO auth** (main.py:203). Once the tunnel is public, anyone can see every customer plan filename, errors, room counts. Add HTTP Basic / shared-secret header, or block the path at the tunnel. Also XSS: filename/error interpolated unescaped into HTML (main.py:243-248) → `html.escape()`.
2. **Rate limiter is effectively OFF behind Cloudflare Tunnel** (main.py:129, rate_limit.py:54). All public traffic arrives as 127.0.0.1 which is EXEMPT. Read CF-Connecting-IP / X-Forwarded-For; drop localhost exemption in prod.
3. **CORS**: ALLOWED_ORIGINS defaults to localhost:5173 (main.py:107-119). Must set the exact Vercel origin in server/.env or nothing works.
4. **VITE_API_BASE** must be set at Vercel build time to the tunnel URL, else the app white-screens by design (scene.ts:4-12, visualize.ts:3-11).
5. **No client timeout/cancel on renders** (api/visualize.ts): renderImage/animateImage fetch with no AbortSignal → stalled GPU = UI hangs forever at "Rendering… Ns". Add AbortSignal.timeout + Cancel button. (scene.ts already does this right — copy the pattern.)
6. **Unbounded RAM read when STORE_UPLOADS=1** (main.py:553): corpus_bytes = await image.read() has NO size cap and runs BEFORE _read_capped. Multi-GB upload = memory DoS. Same for /visualize/render:412 and /visualize/animate:450 (no cap, no type check). Move caps before any .read().
7. **One GPU, two independent semaphores** (main.py:269 _INFER_SLOTS vs visualize.py:36 _SLOTS): an SDXL render + an ML parse can run at once → 12GB OOM → user-facing 503s. Share one global semaphore. (small-medium)
8. **Licensing gates before anything paid** (policy, not code): CubiCasa = CC BY-NC (demo only), tf2/DeepFloorplan = GPL (copyleft — NOT the "commercial-safe" fallback the code comments claim), SVD = Stability community license (revenue cap), seg CN = community checkpoint. Free public beta is defensible; charging money is not. Add env flags to hard-disable NC paths.
9. **Relabel two numbers** (copy change now, math fix later):
   - Super built-up applies loading to BUILT-UP not CARPET (area_statement.py:70) → super inflated, efficiency understated (~77% shown as ~68%). Industry convention is super = carpet × (1+loading).
   - BOQ headline is masonry+finishes only (~half of real build cost) — label must say so unmistakably (boq.py, App.tsx:716).

## B. BIGGEST VALUE LEAKS (this week — these answer "are we extracting max ML value?" → NO, ~60%)

10. **tf2 reader can never emit a door** (tf2_floorplan.py:110-115 builds a 0/1 mask; openings.boxes_from_mask keys door on pixel value 2 which never occurs) → every tf2 scene gets the no_doors −100 penalty → tf2 essentially never wins best-of. The whole second reader is dead weight right now. Fix: small.
11. **Winner-takes-all discards half the ML output** (main.py:498-526 + plan_health.better_scene): loser reader's doors/windows thrown away; no union/voting. Also pays 2× inference for one result.
12. **CubiCasa's room-TYPE map (12 classes) and icon map (toilet/sink/bathtub/closet…) are computed then discarded** (perception.py: detections() uses only Wall + window/door icons). ML-path scenes get untyped rooms and no furniture while the vector path gets both. scene_from_segments has no room-type parameter — that's the plumbing to add.
13. **Seg/depth ControlNet mostly not fed**: no server-side depth/seg render exists; backend only uses maps the FRONTEND uploads, else falls back to Canny of a screenshot (visualize.py:265-313). Seg silently dropped if the 5GB model isn't cached (L276), and loads fp32 (worst case for 12GB). Frontend gbuffer.ts DOES generate maps — verify they actually arrive on every render.
14. **plan_health rewards over-detection** (score = rooms*2 + doors − 100*flags): a hallucinating reader outscores an accurate one. Add plausibility penalties (room min area, doors-per-room bounds).
15. **Free render speed being left on the table** (visualize.py): no xformers/SDPA attention, default scheduler at 28 steps (DPM++ 2M Karras ≈ same quality at ~20-22), cpu_offload left ON even for single-CN fp16 which fits in 12GB (2-3× latency for nothing). ~30-40% faster renders, small effort.
16. **Every hard PDF is parsed up to 3× per request** (main.py:439-485) + /scene and /scene.glb EACH re-run the full parse and queue behind the 1-slot semaphore (scene.ts:79-82 Promise.all) → slow plans take ~2× as long as needed. Cache parse by upload hash.

## C. ACCURACY RISKS (corpus-gate any fix — standing rule 6)

17. Scale from "dominant column = 12 inch" assumption (pdf_vector._scale_ppf:578) — a 9"/230mm column plan rescales everything ~1.3×; _door_scale_sanity only warns (needs ≥5 doors), never corrects. Add: median-door-width cross-check that flags needs_review.
18. assumed_width fallback silently guesses 40ft (scene_builder.DEFAULT_WIDTH_FT) — every downstream ₹/sqft number wrong; only −15 quality points. Surface prominently in UI.
19. rooms[:16] cap (pdf_vector:1258) silently drops rooms on big plans; OCR label fallback only fires at ZERO live labels (1282); Plan-B window detection skipped if even one window found (1351). Make additive.
20. Silent except-pass in is_vector_plan (main.py:493), OCR recovery, cad_vector flatten, and all four analysis blocks (main.py:571-590) — scene can ship with missing vastu/boq/area and no signal. Log + degraded-quality flag.

## D. SUPABASE / EDGE CASES — mostly GOOD

Confirmed good: missing keys degrade gracefully everywhere; fire-and-forget inserts never block (5s/20s timeouts); service key never reaches frontend; RLS on; no PostgREST injection in /review; db.py inserts match schema.sql exactly; 0-byte/fake-PDF/password-PDF/500-page uploads all rejected cleanly.
Gaps: bare asyncio.create_task refs can be GC'd → occasional silently-lost log rows (db.py:141); no unique constraint on scene_hash → duplicate rows on re-upload; parses table + plans bucket + .render_cache all grow forever (no TTL/eviction); rate-limit buckets per-process if you ever run workers>1.
