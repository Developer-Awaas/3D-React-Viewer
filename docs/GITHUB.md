# Pushing this project to GitHub (run on your Windows machine)

## 0. One-time: clean the broken .git (from the earlier sandbox attempt)
PowerShell, in the project folder:
```powershell
cd "$env:USERPROFILE\OneDrive\Desktop\3D React Viewer"
Remove-Item -Recurse -Force .git   # only if a .git folder exists
```

## 1. Create an empty repo on github.com
New repository -> name it (e.g. "drishti-3d") -> **do NOT** add a README/.gitignore/license
(keep it empty so there's nothing to merge).

## 2. Init, commit, and push
```powershell
git init
git config user.name  "Dev_Awaas"
git config user.email "awaas.ai.dev@gmail.com"
git add .
git commit -m "Drishti 3D sandbox: viewer + room/toilet builder + pipeline"
git branch -M main
git remote add origin https://github.com/<your-username>/drishti-3d.git
git push -u origin main
```

## 3. Authentication (important)
GitHub no longer accepts your password over HTTPS. When it asks for a password, paste a
**Personal Access Token (PAT)** instead:
GitHub -> Settings -> Developer settings -> Personal access tokens -> Fine-grained ->
Generate (give it "Contents: read/write" on this repo). Use that token as the password.
(Or install GitHub CLI `gh auth login`, or set up an SSH key.)

## 4. After future changes
```powershell
git add .
git commit -m "short, clear message"
git push
```

---

## The rules (do / don't)
**DO**
- Commit often, with short clear messages ("add kitchen room", "fix toilet door").
- Keep `.gitignore` excluding `node_modules/` and `dist/` (already set) — never commit those.
- Commit source + small assets (your `.glb` files and `public/` images are fine, they're small).
- Pull before you push if others also push: `git pull --rebase`.

**DON'T**
- Never commit secrets — API keys, tokens, passwords, `.env` files. If you add a `.env`,
  put `.env` in `.gitignore` first.
- Don't commit `node_modules/` (huge, regenerated by `npm install`).
- Don't commit the broken `.git` or large binary models (>100 MB) — GitHub rejects >100 MB files.
- Don't force-push (`git push -f`) to a shared branch unless you know why.

## Tip
Your repo already has a `.gitignore` excluding node_modules/dist, and `docs/` with CONTEXT,
GUIDE, CONVERTER, REQUIREMENTS, DESIGN_RULES — so the repo will be clean and self-documenting.
